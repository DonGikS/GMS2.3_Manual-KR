명령 줄 매개 변수

게임을 만들 때 실행 파일을 테스트하여 최종 컴파일이 올 바르고 예기치 않은 오류나 차이가 없는지 확인해야합니다. 그러나 실행 파일을 만들면 디버그 콘솔이나 컴파일러 창에 액세스 할 수 없으므로 발견 할 수있는 문제를 찾아 내고 디버깅하기가 어려울 수 있습니다. 그러나 이것에 대한 수정이 있으며, 명령 행 매개 변수 를 사용하여 게임의 실행 가능 버전이 실행될 때 특정 특성을 설정하여 출력 로그 파일을 작성하고 다른 특수 상황을 테스트 할 수 있습니다.

참고 : 이것은 표준 Windows 대상 모듈 에서만 사용할 수 있습니다.

To get this working you must first create a Windows executable of your game and have it installed on your test machine. Once you have that done, you need to create a Shortcut to the game on the desktop. This is where we will add in the parameters to be passed to the *.exe (to create a shortcut, right click 게임 실행 파일을 선택한 다음 보내기 > 데스크탑 (바로 가기 생성) 을 선택하십시오.

바로 가기를 만든 후에 대상 경로에서 다음 매개 변수 중 하나를 사용할 수 있습니다.

-noaudio : 게임의 모든 오디오를 끕니다.

-inawindow : 전체 화면 모드로 실행되도록 설정 한 경우에도 게임이 창에서 시작되도록합니다.

-output < filename > : 코드베이스에있는 디버그 메시지를 포함하여 콘솔 출력을 파일 이름으로 보냅니다.

-debugoutput < filename > : 사용자 정의 디버그 메시지는 제외하고 버그보고를 위해 러너로부터 추가 정보를 포함하여 콘솔 출력을 파일 이름으로 보냅니다.

-software : 게임 그래픽을 렌더링하기 위해 하드웨어 대신 Software Vertex Processing을 사용합니다. 이는 내장 인텔 GFX 칩셋을 사용하거나 구형 컴퓨터에서 디스플레이 문제를 표시하는 PC에 특히 유용합니다.

-intel : 사용중인 GPU에 관계없이 잘못된 인텔 드라이버에 대한 수정을 켭니다. 이러한 나쁜 드라이버에 대한 해결 방법은 게임 속도가 느려져 게임 성능이 현저하게 저하되므로 수정이 필요하지 않은 경우에는 사용하지 마십시오.

-vanillaGFX :이 기능을 사용하면 드라이버 제조업체에 대한 모든 검사가 해제되며 "있는 그대로"게임을 테스트하는 데 매우 유용 할 수 있습니다.

-output 및 -debugoutput 의 < filename > 은 파일이 각 쓰기마다 열리고 닫히므로 원하는 경우 동일 할 수 있습니다.

이제 게임 실행 파일에 대한 바로 가기를 만든 위치로 이동 한 다음 마우스 오른쪽 버튼을 클릭하고 속성을 선택하십시오.

위와 비슷한 창이 열립니다. 여기서 명령 행을 대상 경로에 추가 할 수 있습니다. 주어진 경로의 끝으로 스크롤하고 공백을 추가 한 다음 매개 변수 문자열을 추가하십시오. 예를 들어 디버그 파일 출력을 만들려면 다음과 같이됩니다.

D:\Users\Me\GameMaker\CustomLoadBar-Default-1.0.0.0\CustomLoadBar.exe -debugoutput debug.txt

이제 완성 된 게임의 디버그 및 컴파일러 정보와 함께 실행 파일과 같은 위치 (바로 가기 위치가 아니라 게임이 설치된 위치)에 " debug.txt "라는 파일이 생성됩니다. "확인"을 클릭 한 다음 바로 가기를 사용하여 게임을 실행하여 이러한 매개 변수를 사용하고 디버깅에 도움이되도록하십시오.

게임을 디버깅하는 방법에 대한 자세한 내용은 디버깅 매뉴얼 섹션을 참조하십시오.

 

명령 줄을 사용한 A / B 테스트 명령 줄을 사용한 A / B 테스트

YoYo 러너는 다음과 같은 특수 명령 줄 매개 변수의 이점도 제공합니다.

-game < filename > : 주어진 게임 파일을로드합니다

왜 이것을 사용 하시겠습니까? 일반적으로 그렇지는 않지만 한 번에 여러 버전의 게임을 실행하려는 경우이를 사용하여 A / B 비교 테스트를 훨씬 쉽게 수행 할 수 있습니다. 이 방법을 사용하면 다른 사람과 함께 테스트에 집중하거나 디자이너로서 병행 테스트를 수행하여 더 나은 느낌을 얻을 수 있습니다.

이를 설정하려면 미리 몇 가지 사항을 알아야합니다. 시작하려면 러너 자체의 경로가 필요합니다. 이 파일은 Windows OS의 Program Data 폴더에 있습니다 (기본적으로 숨겨져 있으므로보기 속성을 변경해야합니다).

C:\ProgramData\GameMakerStudio2\Cache\runtimes\<runtime_version>\windows\Runner.exe

runtime_version 폴더는 현재 설치 한 런타임에 따라 다릅니다. 또한 < yourgame > .win 파일의 경로가 필요합니다. 이것이 러너를 가리키게하기 때문입니다. 가장 쉬운 방법은 Temp 폴더 (일반적으로 % localappdata % \\ Temp \\ GameMakerStudio \\ <game_name> \\에 있음 )로 이동하여 * .win을 검색하는 것입니다. 사용 가능한 모든 파일이 표시되므로 간단히 전체 경로를 얻는 데 필요한 것을 선택하십시오.

이제 실행 파일에 대한 것처럼 러너에 대한 바로 가기를 만든 다음 필요에 따라 매개 변수를 변경할 수 있지만 테스트간에 매개 변수를 열고 편집해야 함을 의미 하므로이 기능을 나란히 사용하는 목적을 상실합니다. 측 테스트. 대신이 문제를 처리하기 위해 몇 개의 작은 "일괄 처리"파일을 만들려고합니다.

이렇게하려면 먼저 바탕 화면에 새 텍스트 문서를 만들고 메모장 (또는 다른 텍스트 편집기)에서 엽니 다. 다음을 추가하십시오.

START C:\ProgramData\GameMakerStudio2\Cache\runtimes\<runtime_version>\windows\Runner.exe -game <LOCAL TEMP PATH>\<GAMENAME>.win

게임 * .win 파일을 가리 키도록 임시 경로를 설정 한 다음이 파일을 " GM_Test_1.bat "로 데스크탑에 저장해야합니다 ( "모두를 선택하지 않으면 메모장이 * .txt 파일로 자동 저장됨에 유의하십시오"). 대화 상자 옵션에서 최종 명령의 예는 다음과 같습니다.

START C:\ProgramData\GameMakerStudio2\Cache\runtimes\runtime-9.9.1.199\windows\Runner.exe -game D:\Users\Mark\AppData\Local\Temp\GameMakerStudio2\AlphaDog_Reboot\AlphaDog_Reboot.win

Now, create another text file and add the same line, only this time you point it to the path for the second version of the game that you want to do side-by-side testing with, and you would save this as "GM_Test_2.bat". You can double click 둘 중 하나가 다른 버전의 게임을 불러오고 배치 파일로 만들면 쉽게 다시 열어서 편집하고 테스트중인 게임을 변경할 수 있습니다.
