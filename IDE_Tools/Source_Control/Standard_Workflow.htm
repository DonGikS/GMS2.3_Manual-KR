표준 워크 플로우

프로젝트를 진행하면서 자연스럽게 스프라이트 생성, 스크립트 편집, 타임 라인 삭제 등 시간이 지남에 따라 프로젝트가 개발하는 데 필요한 모든 것을 얻을 수 있습니다. 이것들은 당신이 추적하고 무언가 잘못되었을 때 되돌아 갈 수있는 모든 행동입니다. 이것이 사고를 되돌릴 수 있도록 소스 제어를 사용하는 주된 이유이며, 처음부터 기본 워크 플로를 신속하게 안내하여 이것이 어떻게 작동하는지 이해할 것입니다.

Starting from the beginning, create a new GML project and then activate source control through the main options. This gives us access to the Source Control menu on the top, so we'll immediately go and select Create Project Repository. If you look at the room resources, you can see that the default room has gained a red exclamation , 마지막 상태에서 수정 되었음을 의미합니다. 프로젝트에 다른 리소스가있는 경우 아래 이미지와 같이 모두 동일한 아이콘이 표시됩니다.

이제 빈 프로젝트를 그대로 커밋하거나 기본 상태로 만드는 옵션이 있습니다. 지금은 첫 번째 커밋을 수행 할 것이므로 소스 제어 > 커밋 변경을 클릭하십시오. 이제 세 개의 영역으로 분할 된 새 창이 열립니다.

  • 단계적 변경 : 소스에 적용 할 변경을 나타냅니다. 이것은 비어있을 수 있지만 주 옵션 변경이 표시 될 수 있습니다.
  • Unstaged Changes : 변경 된 파일을 보여 주지만 Git에 커밋하고 싶다고 말하지 않았으며 변경 사항이 "보류 중"입니다.
  • 커밋 메시지 : 변경 사항을 설명하기 위해 추가 할 수 있습니다.

For now, click the Stage All button, as we want to commit all the changes, and type something like "First Commit!" into the Commit Message and click Commit. All our resources should now have a green tick수정 사항이 없음을 의미합니다.

그것은 당신이 창 내에서 작업 할 때, 다양한 단계적와 Unstaged 메시지가 상징 접두어됩니다 주목할 가치가있다. 사용되는 기호는 작업 및 파일에 따라 달라지며 다음과 같이 사용됩니다.

  • [...] -단계적 변경 (이전 [A])
  • [+] -새로 추가
  • [->] -이름 변경 / 이동
  • [-] -제거
  • [M] -비 단계적 변경
  • [?] -추적되지 않은 파일
  • [!] -누락 된 파일

이제 우리의 작업 프로세스가 시작될 수 있으며 위의 단락에서했던 것처럼 프로젝트를 빌드하고 변경 사항을 저장하고 커밋 할 수 있습니다.

얼마 동안 일한 후에는 언제든지 커밋 한 내용을 확인하고 확인하여 커밋 기록 을 열 수 있습니다. 히스토리 창을 열려면 소스 제어 > 히스토리보기 로 이동하여 세 개의 창이있는 다른 창을여십시오.

맨 위 분할 창은 확약을 설명하고 중간 분할 창은 선택된 확약에 지정된 주석을 설명하고 아래쪽 분할 창은이 확약에서 수정 된 파일을 설명합니다. Diff Tool 을 설정 한 경우이 창에서 파일을 두 번 클릭하면 diff 도구가 열리고 파일 간 변경 사항을 볼 수 있습니다.
